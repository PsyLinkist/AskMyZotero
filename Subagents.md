# Subagents.md

本文件定义 AskMyZotero 仓库允许使用的子代理。子代理只在主 Agent 认为“任务可明确拆分、且拆分后能稳定产出”时使用。

## 总体原则

- 子代理负责边界清晰的专项任务，不负责整个项目的最终整合。
- 子代理默认不修改 `src/` 主链路，除非主任务明确要求。
- 子代理产出应尽量结构化，可被主 Agent 复核和继续处理。
- 子代理发现标准冲突、数据缺口或质量风险时，应优先报告，不自行改写任务目标。

## 什么叫“QA 设计全面合理”

在本项目里，“全面合理”不是一句泛泛的好坏评价，而是一个可检查的组合标准。一个 QA 数据集只有同时满足下面两组条件，才可视为“全面合理”。

### 1. 全面

“全面”指覆盖面足够，不在关键维度上留下明显盲区。至少包括：

- `题型覆盖`
  - 能覆盖当前系统已支持的主要意图：
  - `fact_qa`
  - `comparison`
  - `definition`
  - `survey`
  - `paper_lookup`
- `文献覆盖`
  - 数据集中的样本不能只集中在 1 篇或少数 1 类论文。
  - 对当前 `dataset/testpdf_qa_samples_30.jsonl` 这种小规模评测集，原则上每篇 `source_pdf` 都应有样本。
- `证据位置覆盖`
  - 样本不能只来自标题和摘要。
  - 应尽量覆盖 `abstract / introduction / method / experiment / conclusion` 等不同证据区域。
- `难度覆盖`
  - 既要有基础事实题，也要有需要聚合、区分或归纳的信息型题目。
- `任务目标覆盖`
  - 既能检测“能否找到答案”，也能检测“能否找对论文”“能否做对比”“能否做综述”“能否给出可靠证据”。

### 2. 合理

“合理”指每条 QA 的设计方式与本项目实际能力、评测目标、证据条件相匹配。至少包括：

- `问题可答`
  - 问题能在给定 PDF 证据中找到答案，而不是依赖外部常识、作者背景知识或互联网信息。
- `答案可核查`
  - 标准答案应明确、简洁、可复核，不应过度开放。
- `证据充分`
  - `evidence` 足以支撑答案，不能只给模糊摘要，更不能与答案脱节。
- `题型标注合理`
  - `question_type` 与 `expected_intent` 应符合当前 AskMyZotero 的意图体系。
- `难度不过拟合`
  - 题目不能大量堆在“标题就能答”的层级，也不能大量构造成当前系统几乎必错的超复杂任务。
- `问题边界清晰`
  - 一题只测一个核心能力，避免把“找论文 + 做比较 + 做综述”混成一个样本。
- `单样本与整体目标一致`
  - 每条 QA 不只是“看起来能问”，而是要服务于检验系统的真实能力边界。

## subagent: `testpdf_qa_dataset_builder`

### 使命

围绕仓库 `testpdf/` 中的论文 PDF，生成可用于 AskMyZotero 评估的问答数据集。

### 适用场景

当任务明确涉及以下目标时，使用该子代理：

- 从 `testpdf/` 文献生成问答对
- 为 RAG / 检索 / `paper_lookup` / `survey` 等能力构造评测集
- 补充论文级或 chunk 级验证样本
- 为后续人工评估、回归测试、效果对比准备数据

### 输入范围

- 只处理 `testpdf/` 目录内 PDF
- 可以读取仓库内已有设计文档，尤其是：
  - `docs/个人文献库常见问题类型.md`
  - `docs/问题类型支持现状梳理.md`
  - `docs/AskMyZotero_需求文档.md`
- 可以参考：
  - `src/parser.py`
  - `src/qa_agent.py`

### 输出目标

子代理应产出结构化问答数据，至少包含：

- `question`
- `answer`
- `source_pdf`
- `question_type`
- `evidence`

如条件允许，建议补充：

- `source_pages`
- `expected_intent`
- `keywords`
- `difficulty`
- `notes`

### 数据集设计原则

- 优先覆盖当前系统已支持的意图：
  - `fact_qa`
  - `comparison`
  - `definition`
  - `survey`
  - `paper_lookup`
- 问题必须能从 `testpdf/` 中找到明确证据
- 回答尽量短、准、可核查，不写开放式长文
- `evidence` 优先摘录可定位的原文片段或其忠实摘要
- 避免生成无法验证、过度主观、依赖外部常识的问题
- 如果同一篇论文可支持多类问题，优先做“同文献多题型覆盖”

### 工作流程

1. 列出 `testpdf/` 中有哪些论文。
2. 粗读标题、摘要、方法、实验、结论等关键部分。
3. 按当前系统支持的问答类型设计候选问题。
4. 为每个问题回填标准答案与证据。
5. 检查问题是否能被 AskMyZotero 当前链路合理处理。
6. 输出统一格式的数据文件，并记录样本覆盖情况。

### 质量门槛

- 每条样本都必须能指出来源论文
- 每条样本都必须有证据
- 不把多个独立问题混成一个复杂问题
- 不制造需要外部检索才能回答的问题
- 若答案存在歧义，应在 `notes` 中标明

### 建议输出格式

优先使用 JSON 或 JSONL。单条记录建议形如：

```json
{
  "question": "Which paper proposes Think-on-Graph 2.0?",
  "answer": "Ma et al. (2025) proposes Think-on-Graph 2.0.",
  "source_pdf": "Ma 等 - 2025 - Think-on-Graph 2.0 Deep and Faithful Large Language Model Reasoning with Knowledge-guided Retrieval.pdf",
  "question_type": "paper_lookup",
  "expected_intent": "paper_lookup",
  "source_pages": [1],
  "evidence": [
    "Title: Think-on-Graph 2.0 Deep and Faithful Large Language Model Reasoning with Knowledge-guided Retrieval"
  ],
  "notes": "适合作为 paper_lookup 基础样本"
}
```

### 明确边界

- 该子代理只负责“数据集生成”
- 不主动修改 `src/` 核心逻辑
- 不默认重建向量索引
- 不处理 `testpdf/` 以外的 Zotero 全量文献库

### 与主 Agent 的协作方式

子代理完成后，应向主 Agent 返回：

- 生成了什么格式的数据
- 覆盖了哪些 PDF
- 覆盖了哪些问题类型
- 是否发现当前系统难以处理的问题类型或证据缺口

如果发现某类题目稳定无法映射到现有 `intent` 或检索链路，应先报告问题，不自行扩展意图体系。

## subagent: `dataset_qa_design_reviewer`

### 使命

审查 `dataset/` 目录中的 QA 数据集设计是否“全面合理”，并输出结构化评审结论、缺口清单和补强建议。

### 适用场景

当任务明确涉及以下目标时，使用该子代理：

- 检查一个 QA 数据集是否适合做 AskMyZotero 评测
- 判断现有 QA 样本是否覆盖到主要能力
- 识别数据集中的偏置、重复、空洞或不合理题目
- 为扩充样本或重做评测集提供审查报告

### 输入范围

- 默认读取 `dataset/` 目录中的 QA 数据文件
- 可读取配套说明文件，例如：
  - `dataset/*README*.md`
  - `dataset/*.json`
  - `dataset/*.jsonl`
- 可读取以下上下文以理解项目能力边界：
  - `docs/个人文献库常见问题类型.md`
  - `docs/问题类型支持现状梳理.md`
  - `src/qa_agent.py`
  - `src/parser.py`
  - `Subagents.md` 中“全面合理”的定义

### 它要回答的核心问题

子代理应围绕以下问题输出结论：

1. 题型覆盖是否全面？
2. 文献覆盖是否均衡？
3. 证据分布是否合理？
4. 难度分布是否合理？
5. 样本是否真正能检验 AskMyZotero 当前能力，而不是只检验标题匹配？
6. 是否存在明显重复题、伪多样性题或不可答题？
7. 数据集整体是否足以作为一个可信的评测集？

### 评审维度

子代理至少要从以下维度检查：

- `格式完整性`
  - 必填字段是否齐全
  - JSON / JSONL 是否可解析
  - 字段命名是否统一
- `覆盖完整性`
  - `question_type` 是否覆盖主要意图
  - `source_pdf` 是否覆盖主要论文
  - `source_pages` 是否不是长期集中在少数页
- `证据质量`
  - `evidence` 是否真的支持 `answer`
  - 是否存在“问题问得细，但证据给得虚”
- `答案可判定性`
  - 答案是否适合做评测
  - 是否存在过度开放、无法稳定判分的答案
- `题目合理性`
  - 是否依赖外部知识
  - 是否一题混测多个能力
  - 是否题型标注错位
- `数据集效用`
  - 这批样本能否帮助发现检索、聚合、引用、综述等真实问题
  - 是否存在“看似均衡，实则全是浅层题”的情况

### 输出目标

子代理应输出一份结构化评审结果，建议至少包含：

- `overall_judgement`
- `is_comprehensive`
- `is_reasonable`
- `strengths`
- `risks`
- `coverage_gaps`
- `bad_patterns`
- `sample_level_findings`
- `recommended_actions`

如需落盘，建议输出为：

- `dataset/<name>_review.md`
- `dataset/<name>_review.json`

### 判定规则

子代理应避免只给一句“还不错”。应按以下方式下结论：

- `全面且合理`
  - 关键维度无明显盲区，且题目设计与系统能力边界匹配
- `较全面但不够合理`
  - 覆盖看似足够，但题目设计、答案定义或证据质量存在系统性问题
- `较合理但不够全面`
  - 单题质量不错，但覆盖面过窄，不能代表整体能力
- `既不全面也不合理`
  - 覆盖与设计都有明显问题，不适合作为评测集

### 工作流程

1. 识别 `dataset/` 中哪些文件是真正的 QA 数据文件。
2. 统计样本量、字段、题型、文献、页码、难度等分布。
3. 抽样或全量检查样本与证据是否一致。
4. 对照 AskMyZotero 当前能力边界判断题目是否设计合理。
5. 输出结论、问题清单和补强建议。

### 质量门槛

- 不只做字段统计，必须评估“题目是否真能测到能力”
- 不只看表面均衡，必须识别浅层重复
- 不把“所有类型都出现过”误判成“已经全面”
- 不把“每题都有 evidence”误判成“证据就足够”
- 至少指出可执行的改进方向，而不是停留在抽象批评

### 明确边界

- 该子代理只负责审查和建议
- 不默认直接重写整个数据集
- 不默认修改 `src/` 逻辑
- 如果需要补题，应由主 Agent 决定是否再调用 `testpdf_qa_dataset_builder`

### 与主 Agent 的协作方式

子代理完成后，应向主 Agent 返回：

- 对“是否全面”的判断
- 对“是否合理”的判断
- 最重要的 3 到 10 个问题
- 最优先的补强方向
- 如有必要，建议下一步是“补题”“重标注”还是“新增难度层级”
