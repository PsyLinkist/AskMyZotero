# 任务 2 与任务 4 待完成事项整理

基于当前 AskMyZotero 后端代码的实际完成度来看，**任务 2（实现 Agent 模块的查询解析）** 和 **任务 4（实现前端与 Agent 的交互）** 还没有真正完成。

目前后端已经具备的是：

- PDF 解析与切块：`parser.py`
- 向量库构建 / 加载：`indexer.py`
- RAG 检索与生成回答：`indexer.py`
- 配置管理与主流程串联：`config.py`、`main.py`
- 快照记录：`manifest.py`

也就是说，当前更准确的状态是：

> **后端已经实现了 RAG 核心能力，但还没有完成独立的 Agent 模块封装，也还没有完成面向前端的标准交互接口。**

## 任务 2：实现 Agent 模块的查询解析

当前还没有真正独立的 Agent 模块，因此这一部分至少还需要补充以下内容。

### 1. 新增独立的 Agent 层

需要单独新增一个 `src/agent.py`，不要让前端或主程序直接调用 `indexer.py`。

建议后端调用链改为：

```text
用户问题 → Agent 解析 → Agent 调用 RAG / 检索 → 返回结构化结果
```

---

### 2. 实现查询解析函数 `parse_query()`

需要新增一个独立的查询解析函数，用于对用户原始问题做基础分析。它至少需要完成：

- 接收用户原始问题
- 提取关键词
- 判断问题类型
- 判断是否是文献检索类、总结类、对比类、作者/年份/方法类问题
- 生成结构化解析结果

例如可以返回如下结构：

```python
{
    "raw_query": "哪些文章提到了 GraphRAG？",
    "intent": "paper_search",
    "keywords": ["GraphRAG"],
    "filters": {},
    "need_rag": True
}
```

---

### 3. 实现 Agent 主入口函数

需要在 Agent 中补一个统一入口，例如：

- `run()`
- `handle_query()`

这个入口函数负责根据“解析结果”把请求路由到后续能力上，比如：

- 普通文献问答 → 调用 RAG
- 只想查文献标题/来源 → 只调检索，不必生成长回答
- 后续若加入 Zotero 元数据查询，也可以在这里进行分流

---

### 4. 定义统一输出格式

当前 `indexer.py` 更偏向命令行使用，很多内容是直接 `print()` 出来的。

但 Agent 模块对外时，应该返回标准化结果对象，而不是只在终端里打印。建议输出结构至少包含：

```python
{
    "query": "...",
    "intent": "...",
    "answer": "...",
    "references": [
        {
            "source": "...",
            "page": 3,
            "score": 0.82
        }
    ]
}
```

---

### 5. 将后端能力统一收口到 Agent

后续前端不应再直接碰下面这些底层模块：

- `scanner.py`
- `parser.py`
- `indexer.py`

而应该统一通过 Agent 层暴露出的接口进行调用。

因此，任务 2 本质上是：

> **在现有 RAG 能力之上，再增加一层“查询解析 + 路由调度 + 统一输出”的 Agent 外壳。**

---

## 任务 4：实现前端与 Agent 的交互

这一部分目前基本也还没有真正完成，因为当前系统仍然主要是命令行模式。

### 1. 确定前后端交互方式

你们之前计划采用**直接函数调用**进行模块间通信，那么当前需要提供一个给前端直接调用的函数接口，例如：

```python
agent.handle_query(user_query: str) -> dict
```

前端只需要将输入框中的查询文本传入，再接收一个结构化结果对象用于展示。

---

### 2. 将命令行输入输出改为函数输入输出

当前很多流程仍然基于：

- `input()`
- `print()`
- 流式打印回答

这些都不适合直接与前端对接。

因此需要逐步改成：

- 输入：通过函数参数传入
- 输出：通过结构化字典或对象返回

---

### 3. 设计前端需要的返回字段

前端展示时，通常至少需要这些字段：

- 用户原问题
- 解析后的问题类型
- 最终回答
- 命中文献列表
- 每条文献的标题 / 路径 / 页码 / 摘要片段
- 是否出错
- 错误信息

建议后端统一返回类似这样的结构：

```python
{
    "success": True,
    "intent": "paper_search",
    "answer": "...",
    "references": [...],
    "snippets": [...],
    "error": None
}
```

---

### 4. 增加中间适配层

前端展示的内容，不一定等于 RAG 的原始输出结果。

因此最好增加一层适配逻辑，把后端结果整理成更适合前端展示的形式，例如拆成：

- `answer`：最终回答
- `references`：文献来源列表
- `snippets`：命中的片段
- `metadata`：作者、年份、文件名等

---

### 5. 加入异常处理与前端友好反馈

当前后端报错很多还是直接在终端中输出。后续需要将这些异常转为前端可识别的状态和提示。

例如：

- 没有找到相关文献
- API Key 无效
- 向量库未构建
- Zotero 路径错误
- 模型接口超时

这些可以返回一个统一的错误对象，例如：

```python
{
    "success": False,
    "error": {
        "code": "NO_RESULTS",
        "message": "没有找到相关文献，请尝试其他关键词。"
    }
}
```

---

### 6. 与前端约定输入输出协议

需要尽快与前端同学约定好接口格式。

例如前端发给后端：

```json
{
  "query": "哪些文献提到了时间性知识图谱？"
}
```

后端返回：

```json
{
  "success": true,
  "intent": "paper_search",
  "answer": "...",
  "references": [...],
  "snippets": [...]
}
```
