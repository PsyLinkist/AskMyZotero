# 问题类型支持现状梳理

本文用于整理 `个人文献库常见问题类型.md` 中各类问题在当前项目代码中的支持现状，区分：

- 当前最能应对的问题
- 当前可以应对，但能力还不够专门的问题
- 当前明显不足、暂时无法可靠应对的问题

本文基于当前实际生效的主链路整理，而不是基于设计文档中的未来方案。

---

## 一、当前实际生效的能力基础

当前项目已经接入并实际生效的核心能力主要包括：

- 意图识别：`fact_qa / comparison / definition / survey / paper_lookup`
- LLM 驱动的 query rewrite
- section-aware chunking
- FAISS dense retrieval
- 基于作者 / 标签 / 文集 / 年份的硬过滤
- chunk 检索结果按论文聚合
- 证据片段、页码、引用编号输出

当前主链路大致为：

`query -> 意图识别 -> query rewrite -> 元数据硬过滤 -> FAISS 检索 -> 去重合并 -> 论文级聚合 -> 回答生成`

因此，当前系统并不是为 12 类问题分别实现了 12 套检索策略，而是用 5 类粗粒度 intent 和一套通用检索链路去覆盖这些问题。

---

## 二、当前最能应对的问题

这几类问题与当前代码的能力最贴合，理论上也是当前系统表现最稳定的部分。

### 1. 概念解释类

支持度：高

原因：

- 已有 `definition` intent
- section 权重偏向 `abstract / introduction`
- 适合返回直接回答 + 少量代表论文

为什么适合：

- 这类问题本质上需要找到概念定义、术语解释、方法核心思想
- 当前系统能优先利用摘要、引言等更适合定义型问题的区块

典型适用问题：

- 什么是 HippoRAG？
- GraphRAG 的核心思想是什么？

---

### 2. 方法使用类

支持度：高

原因：

- 当前 query rewrite 对这类问题有明显针对性
- 会尽量把“和某方法相关”改写为“在哪些论文中使用了该方法”
- `paper_lookup` 输出形式与这类问题高度匹配
- 能输出论文列表和使用证据

为什么适合：

- 当前系统最接近“找哪些论文把某方法作为组件使用”的目标
- section 权重也会照顾 `method / abstract / experiment`

典型适用问题：

- 哪些文章使用了 PPR 算法？
- 哪些工作采用了 contrastive learning？

备注：

- 这是当前项目最值得保留和继续强化的一类问题

---

### 3. 主题找文献类

支持度：高

原因：

- `paper_lookup` 和 `survey` 都能覆盖“找相关论文”场景
- 当前系统本身就比较擅长输出论文列表
- 可以结合作者、标签、文集、时间范围做候选过滤

为什么适合：

- 这类问题主要追求相关论文召回，而不是严格的结构化分析
- 当前 dense retrieval + 论文聚合基本能胜任

典型适用问题：

- 我读过哪些关于 multi-hop retrieval 的工作？
- 哪些论文关注 long-context reasoning？

---

### 4. 综述梳理类

支持度：较高

原因：

- 已有 `survey` intent
- section 权重偏向 `abstract / introduction / conclusion`
- 可以对多篇论文做聚合式总结

为什么适合：

- 这类问题需要多论文汇总，而当前系统已经支持论文级聚合和证据整理

典型适用问题：

- 帮我梳理一下文献库里关于 RAG evaluation 的工作
- 给我总结一下这个主题下的主流路线

当前边界：

- 能做多篇论文的综述式总结
- 但还做不到特别强的“自动分组、路线图、脉络树”

---

### 5. 阅读回忆与定位类

支持度：较高

原因：

- 当前系统保留了全文 chunk、页码、section、上下文窗口
- 可以根据模糊描述召回候选论文
- 能返回证据片段帮助用户确认

为什么适合：

- 这类问题不是严格事实判断，而是“帮我把记忆重新定位到论文”
- 当前全文 chunk 检索 + 证据片段展示比较适合这个目标

典型适用问题：

- 我之前看过一篇把超图用于知识检索的论文，是哪篇？
- 那篇说 PPR 可用于局部扩散的论文在哪？

---

## 三、当前可以应对，但还不够专门的问题

这几类问题不是完全做不了，但当前更多是依赖通用检索链路“顺带覆盖”，缺少专门设计。

### 6. 对比分析类

支持度：中

原因：

- 已有 `comparison` intent
- section 权重会偏向 `method / experiment / conclusion`
- 可以生成对比型回答

当前不足：

- 没有真正的结构化对比表
- 没有稳定的“比较对象识别与对齐”
- 很难保证总是按任务、方法、优缺点等维度输出

适用但有限的场景：

- HippoRAG 和 GraphRAG 有什么区别？

---

### 7. 证据定位类

支持度：中

原因：

- 当前可以返回证据片段、页码、引用编号
- 可以做 direct answer，并附证据摘要

当前不足：

- 没有专门的“支持 / 不支持 / 证据不足”判断器
- 没有句级验证或 claim verification 机制
- 复杂判断容易受到召回噪声影响

适用但有限的场景：

- 有没有文献说 long context 会干扰 LLM？

---

### 8. 实验结果类

支持度：中

原因：

- parser 能识别 `experiment / results` 一类区块
- 排序时会对实验相关区块给更高权重
- 可以检索出实验描述与结果片段

当前不足：

- 不能稳定抽取表格中的指标
- 不能结构化比较实验结果
- 很难可靠回答“谁在某指标上最好”

适用但有限的场景：

- 哪些论文在 HotpotQA 上评测过？
- 谁报告了 recall@20？

---

### 9. 数据集与任务类

支持度：中

原因：

- 可以通过摘要、实验部分检索数据集或任务名称
- 可以输出相关论文列表

当前不足：

- 没有专门的 dataset / task 标准化字段
- 没有任务标签体系
- 同义表达、缩写、别名的统一能力较弱

适用但有限的场景：

- 哪些论文做了 multi-hop QA？
- 哪些工作在 MuSiQue 上测试过？

---

## 四、当前明显不足的问题

这几类问题在 `个人文献库常见问题类型.md` 中很重要，但当前代码没有相应的专门方法，回答稳定性会比较差。

### 10. 方法提出类

支持度：低

原因：

- 当前 intent 中没有“方法提出类”这一独立类别
- 当前系统更擅长识别“哪些论文使用了某方法”，不擅长识别“哪些论文首次提出了某方法”
- “提出”和“使用”在现有 `paper_lookup` 中容易混淆

为什么不足：

- 没有 proposal-oriented query routing
- 没有专门强化 `title / abstract / introduction / conclusion` 中“propose / introduce / first define”这类模式

典型困难问题：

- 哪篇论文提出了 HippoRAG？
- 谁最早提出 topic-sensitive PageRank？

---

### 11. 引用与脉络类

支持度：低

原因：

- 当前没有 citation graph
- 没有 references 关系建模
- 没有 related work 依赖关系抽取

为什么不足：

- 这类问题本质上需要“论文之间的关系”，而不只是语义相似检索
- 当前系统只有全文检索和论文聚合，没有真正的文献关系网络

典型困难问题：

- HippoRAG 受哪些工作影响？
- 这几篇论文之间是什么关系？
- 这个方向的方法演进路径是什么？

---

### 12. 写作辅助类

支持度：低

原因：

- 当前可以做多篇论文总结
- 但没有专门的写作组织能力

为什么不足：

- 没有“related work 自动分组”
- 没有“按研究路线整理”
- 没有“生成写作提纲、段落结构、论证骨架”的专门模块

典型困难问题：

- 帮我把这些论文整理成 related work
- 这些文献可以怎么分组？
- 围绕图检索写一个研究综述框架

---

## 五、支持现状总结

可以把当前 12 类问题的支持现状概括为：

### 最能应对

- 概念解释类
- 方法使用类
- 主题找文献类
- 综述梳理类
- 阅读回忆与定位类

### 可以应对，但不够专门

- 对比分析类
- 证据定位类
- 实验结果类
- 数据集与任务类

### 当前明显不足

- 方法提出类
- 引用与脉络类
- 写作辅助类

---

## 六、根本原因

当前项目能覆盖上述问题，但覆盖程度不一致，根本原因在于：

1. 当前只实现了 5 类粗粒度 intent，而不是 12 类真实问题类型。
2. 当前检索主链路仍以通用 dense retrieval 为主，没有对每一类问题配置独立检索策略。
3. 当前系统虽然有 section-aware 权重和论文级聚合，但还没有：
   - proposal-oriented 检索
   - citation / relationship 建模
   - claim verification
   - dataset / metric / task 结构化抽取
   - 写作辅助型输出规划

---

## 七、后续优先建议

如果后续继续沿着 `个人文献库常见问题类型.md` 演进，建议优先顺序如下：

### 第一优先级

- 强化方法使用类
- 强化证据定位类
- 强化主题找文献类
- 强化概念解释类

原因：

- 这几类已经有较好的底座
- 改造成本相对低
- 最容易直接提升用户感知

### 第二优先级

- 增加“方法提出类”独立 intent
- 增加实验结果 / 数据集 / 任务类的结构化识别
- 强化 comparison 的结构化输出

### 第三优先级

- 引入引用关系与文献脉络能力
- 引入写作辅助能力

这两类问题价值很高，但已经超出当前单纯 dense retrieval + paper aggregation 的能力边界。
