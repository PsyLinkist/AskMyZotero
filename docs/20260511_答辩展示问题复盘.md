# 答辩展示问题复盘：检索调优价值与多模态边界

日期：2026-05-11

## 背景

本次答辩后发现两个展示层面的问题。它们不一定说明项目实现失败，但说明项目价值没有被充分表达，尤其是最花时间的检索调优部分没有被转化为清晰、可验证、能说服评审的证据。

这份文档用于后续自我提醒，也作为之后修改论文、PPT、答辩稿和项目路线图时的参考。

## 问题一：没有充分展示检索调优的主要价值

### 答辩中的问题

项目的完整链路已经展示出来了，包括 Zotero 扫描、PDF 解析、元数据补齐、section-aware chunking、FAISS 索引、metadata.db、意图识别、query rewrite、paper-level aggregation 和 citation-grounded answer。

但是答辩表达更偏“系统完整性”，没有充分说明：

- 为什么这些设计能提升检索效果；
- 检索调优具体解决了哪些失败场景；
- 优化前后效果有什么差异；
- 有没有明确指标、样例或实验来支撑“检索效果不错”。

这导致评审可能只看到“做了一个 RAG 系统”，而没有看到项目中最核心、最耗时的价值：围绕个人 Zotero 文献库场景，对检索链路做了多轮工程化调优。

### 应该强调的项目优点

后续展示时，应把检索调优作为项目的核心贡献之一，而不是隐藏在系统流程图里。

可以重点说明：

- `section-aware chunking`：让 chunk 携带 abstract、method、experiment、conclusion 等结构信息，后续排序可以根据问题类型给不同 section 加权。
- `intent-aware routing`：不同问题类型走不同回答策略，例如 paper_lookup 和 fact_qa 的目标不同，不能都按普通 top-k chunk 回答。
- `query rewrite`：针对用户短查询、缩写、方法名和论文术语，扩展出更接近论文正文表达的检索 query。
- `metadata hard filter`：作者、年份、标签、合集等结构化条件不靠 prompt 猜，而是先在 metadata.db 中硬过滤候选范围。
- `paper-level aggregation`：先检索 chunk，再聚合到 paper，避免回答“哪些论文”时只返回碎片化片段。
- `rerank scoring`：结合向量相似度、关键词命中、实体命中、section 权重、页码位置等信号重新排序。
- `citation-grounded answer`：回答必须绑定检索证据，并在无证据时明确拒答或说明证据不足。

### 后续应该补充的指标

后续应尽量用明确指标说明优化效果，例如：

- Retrieval Recall@K：标准答案论文是否出现在 top-k 检索结果中。
- Paper Recall@K：经过 paper-level aggregation 后，目标论文是否进入 top-k paper。
- MRR：目标论文排名是否靠前。
- Citation Accuracy：答案中引用的证据是否真的支持对应结论。
- Answer Groundedness：回答是否只基于检索证据，没有额外编造。
- No-evidence Handling：无相关文献时是否能正确拒答或说明不足。

如果时间有限，也至少应准备“优化前后对比样例”，例如：

- 原始 dense retrieval 只命中标题相关论文；
- 加 query rewrite 后能命中方法 section 中真正使用该技术的论文；
- 加 paper-level aggregation 后，结果从零散 chunk 变成清晰论文列表；
- 加 metadata hard filter 后，按年份、作者、合集筛选更稳定。

### 展示方式建议

PPT 中应单独增加一页或两页：

1. “检索调优是项目核心工作”
   - 列出原始 RAG 的典型问题；
   - 对应展示本项目的改造策略。

2. “优化前后效果对比”
   - 使用 3 到 5 个代表性问题；
   - 展示 baseline、优化后、正确证据、引用结果；
   - 尽量给出 Recall@K、Paper Recall@K、MRR 或人工判定结果。

3. “为什么不是只调 Prompt”
   - 强调很多问题来自检索链路和数据准备；
   - 项目通过 metadata.db、section-aware chunking、aggregation 和 rerank 改善证据质量。

### 可复用表述

后续答辩时可以这样说：

> 本项目的重点不是简单把 PDF 丢进向量库再让大模型回答，而是围绕个人 Zotero 文献库的典型问题，对检索链路进行了专门调优。比如对于“哪些论文使用某个方法”这类问题，系统不能只返回相似 chunk，而要识别这是 paper_lookup，扩展 query，优先关注 method、abstract 等 section，最后把 chunk 聚合回论文级结果。后续我会用 Recall@K、Paper Recall@K、MRR 和引用准确率来量化展示这些优化带来的提升。

## 问题二：没有说明多模态图表处理的取舍

### 答辩中的问题

论文中确实经常包含图、表、公式、流程图和实验结果截图。当前系统主要处理 PDF 文本，没有专门处理图表等多模态内容。

答辩时没有把这个边界说明清楚，容易让评审理解为：

- 项目忽略了论文中的重要信息；
- 系统对图表类问题能力不足；
- 设计时没有考虑多模态数据。

实际上更准确的说法应该是：我们知道图表处理是重要方向，但在当前阶段没有把它作为优先级最高的任务，因为项目目标首先是完成可靠的本地文献文本 RAG，而且大量图表信息通常会在正文、caption 或实验描述中被文本化表达。

### 当前取舍的合理性

后续展示时应明确说明：

- 本项目当前聚焦文本型学术问答，包括方法、结论、相关工作、论文查找、比较和引用式回答。
- 多数图表的核心含义会在论文正文、图表标题、caption 或实验分析段落中出现，因此文本检索可以覆盖一部分图表相关信息。
- 图表解析需要 OCR、版面分析、表格结构恢复、图片理解、公式识别和多模态 embedding，工程复杂度高，短期内会显著增加系统不确定性。
- 在有限时间内，项目优先保证文本链路的准确性、可解释性和可验证性，而不是过早引入不稳定的多模态模块。

### 应承认的局限

也要明确承认当前系统的边界：

- 对扫描版 PDF 支持有限；
- 对只存在于图片中的信息支持有限；
- 对复杂表格、公式、图像实验结果不能稳定结构化理解；
- 如果用户直接问“图 3 展示了什么”“表 2 中哪个方法最好”，当前系统可能只能依赖正文或 caption，不能保证完整读取图表内容。

### 后续改进方向

后续可以把多模态支持拆成几个低风险阶段：

1. Caption-first
   - 优先提取 figure/table caption；
   - 把 caption 作为特殊 chunk；
   - 支持“图 X / 表 X”相关问题的基本检索。

2. Table text extraction
   - 尝试提取 PDF 中可复制的表格文本；
   - 保留行列结构或至少保留表格标题、指标名和数值。

3. OCR fallback
   - 对扫描版或图片型页面引入 OCR；
   - 只作为可选能力，不影响主链路稳定性。

4. Multimodal evidence
   - 对关键图片区域生成描述；
   - 将图片描述、caption、页码和截图坐标一起作为 evidence；
   - 在前端展示时允许用户跳转到原 PDF 页。

5. Evaluation
   - 单独构造图表问答评测集；
   - 不把多模态能力混入普通文本问答指标中，避免评价混乱。

### 可复用表述

后续答辩时可以这样说：

> 当前系统主要处理论文文本内容，没有把图表多模态解析作为本阶段重点。原因有两个：第一，本项目的核心目标是构建可靠的 Zotero 本地文献文本 RAG，优先保证检索、聚合和引用链路稳定；第二，很多图表的核心信息会在正文、caption 或实验分析中被文字描述，因此文本链路已经能覆盖一部分实际问答需求。当然，这也是当前系统的局限：如果问题直接依赖图片、复杂表格或扫描版内容，系统还不能稳定回答。后续我计划先从 caption 提取和表格文本提取做起，再逐步扩展到 OCR 和多模态 evidence。

## 警示

以后展示项目时，不要只讲“系统做了哪些模块”，还要讲“这些模块解决了哪些具体失败场景、带来了什么可观察的效果”。

尤其对于 RAG 项目，评审真正关心的不是流程图有多完整，而是：

- 检索是否更准；
- 证据是否可追溯；
- 回答是否少幻觉；
- 失败边界是否讲清楚；
- 优化是否能被指标或案例证明。
