## 二、元数据筛选改造为两层过滤

### P0：确定统一过滤模型

- [ ] 梳理当前检索链路中“查询改写、召回、重排、问答”的数据流。
- [ ] 定义统一的过滤条件结构，至少覆盖：
  - [ ] `collections`
  - [ ] `tags`
  - [ ] `date_range`
  - [ ] `authors`
- [ ] 区分两类过滤来源：
  - [ ] LLM 自然语言解析出的隐式过滤条件
  - [ ] 前端显式设置的硬过滤条件
- [ ] 明确最终生效规则为“两层条件取交集”。
- [ ] 明确冲突行为：若两层过滤冲突导致无结果，返回空结果而不是报错。

### P0：Collection 筛选

- [ ] 前端展示 collection 筛选控件。
- [ ] 支持三种范围：
  - [ ] 全库
  - [ ] 单个 collection
  - [ ] 多个 collection
- [ ] 前端可展示当前已选择的 collection。
- [ ] 后端按 collection 范围限制检索。
- [ ] 未选择 collection 时，默认表示全库。

### P0：Tag 筛选

- [ ] 前端展示 tag 筛选控件。
- [ ] 支持单个或多个 tag。
- [ ] 后端按 tag 条件限制检索。
- [ ] 如果当前代码已有 tag 过滤语义，则沿用现有规则。
- [ ] 如果当前没有既有规则，则默认实现为“命中任一所选 tag”。

### P0：时间筛选

- [ ] 前端提供时间筛选控件。
- [ ] 至少支持开始时间和结束时间组成的区间过滤。
- [ ] 后端基于现有标准时间元数据字段执行过滤。
- [ ] 统一时间字段映射，避免前后端命名不一致。

### P0：作者筛选

- [ ] 前端提供作者筛选输入或选择能力。
- [ ] 后端支持按作者过滤结果。
- [ ] 若项目已有作者候选、自动补全或标准化逻辑，优先复用。
- [ ] 若没有现成能力，先实现稳定的文本匹配版本。

### P1：两层过滤接入问答链路

- [ ] 在 LLM 解析阶段输出结构化过滤条件。
- [ ] 将 LLM 过滤条件与前端硬过滤条件合并后，再进入检索。
- [ ] 保证召回、重排、最终引用都基于同一份合并后的过滤结果。
- [ ] 记录调试信息，便于确认每次请求实际生效的过滤条件。

### 验收

- [ ] 不选任何硬过滤条件时，搜索范围默认为全库。
- [ ] 选择单个 collection 时，只返回该 collection 内结果。
- [ ] 选择多个 collection 时，只返回这些 collection 并集范围内结果。
- [ ] 设置 tag / 时间 / 作者后，结果只来自符合条件的文献。
- [ ] 用户问题中的自然语言限制与前端硬筛选会同时生效。
