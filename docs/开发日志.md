## 2026-05-05

### 借鉴 RAG-Assistant 的回答 prompt 改造

- 新增 `src/answer_prompts.py`，集中维护学术问答回答模板。
- 将 `paper_list` 和 `direct_answer` 两条回答链路统一到结构化学术助手 prompt：
  - 直接回答
  - 关键证据
  - 综合判断
  - 必要时说明证据边界
- prompt 明确要求只基于给定证据回答，不使用外部知识，不编造论文内容、实验结果或作者观点。
- prompt 要求只引用真正支撑当前句子的证据，避免为了覆盖候选文献而堆叠引用。

### 意图识别 prompt 调整

- 保留 `fact_qa / definition / comparison / survey / paper_lookup` 五类 intent。
- 将 LLM intent prompt 改为按“用户期望的输出形态”分类，而不是按孤立关键词分类。
- 明确区分：
  - `paper_lookup`：用户明确想找、列举、推荐或识别论文/文献。
  - `comparison`：用户要求比较、差异、优劣或 trade-off。
  - `survey`：用户要求综述、概览、主题梳理、趋势或 workflow。
  - `definition`：用户要求解释概念/术语/方法定义。
  - `fact_qa`：用户要求具体机制、过程、结果、数字、证据或实现细节。
- 对中文“如何 / 怎么 / 怎样”和英文 `how does / how is` 类问题补充结构规则，减少机制型问题被误判为论文列表。

### paper 聚合与证据摘录调整

- paper aggregation 的 query terms 现在综合使用：
  - 原始 query
  - entities
  - keywords
  - rewritten queries
- 证据摘录从“取最早命中句”改为按查询词覆盖度和查询词具体度选择句子。
- 移除了针对具体 query 内容的硬编码规则，不再特殊处理 `PPR / PageRank`。
- 适度放宽 paper-level evidence 摘录长度，降低关键句被截断的概率。
- paper context 现在会向 LLM 提供题名、作者、年份、venue、匹配原因和证据片段。

### 引用、格式与 Sources 行为

- 移除旧的“无引用时强行补齐候选引用”和“补充相关文献：[1][2]...”逻辑，避免引用和答案正文对不上。
- 调整回答清洗逻辑，不再删除编号列表、证据标题等结构化 Markdown 内容。
- 修复 `server.py` 中引用重映射后压缩空白时误删 Markdown 段落换行的问题。
- 当前 Sources 面板仍会按正文引用编号过滤和重映射，因此检索/聚合候选数量可能多于前端最终展示数量；后续可考虑区分“候选 sources”和“答案实际引用 sources”。
