# 开发日志

## 2026-04-21

### 硬筛选第一阶段落地

- 前端主页面新增左侧硬筛选 UI，接入：
  - `Date range`
  - `Author`
  - `Tag`
  - `Collection`
  - `Results`
- 移除未启用的 `Archive` 与 `Item type`
- 页面标题与顶部产品名统一改为 `AskMyZotero`
- 前端开始向 `/api/ask` 提交 `filters` 与 `top_k`

### Zotero 元数据接入

- 从 Zotero 本地 sqlite 补充提取：
  - `authors`
  - `tags`
  - `collections`
  - `year/date`
- 元数据从父条目下沉到 attachment，再下沉到 chunk metadata

### Collection 层级展开

- 增加 collection 层级路径展开逻辑
- 若文献位于 `A/B/C`，则支持匹配：
  - `A/B/C`
  - `A/B`
  - `A`
  - `B/C`
  - `B`
  - `C`

### 硬筛选后端链路

- 定义统一硬筛选结构：
  - `collections`
  - `tags`
  - `authors`
  - `date_range`
  - `top-k`
- 作者筛选实现为不区分大小写的模糊包含匹配
- tag / collection 多值筛选实现为命中任一即可
- 时间筛选实现为年份区间过滤

### 元数据库与检索链路改造

- 新增 SQLite 元数据库：
  - `metadata.db`
- 建立表：
  - `papers`
  - `chunks`
  - `paper_authors`
  - `paper_tags`
  - `paper_collections`
- `chunks` 表补充：
  - `chunk_id`
  - `paper_id`
  - `faiss_row_id`
  - `docstore_id`
- 索引加载/重建时同步构建 `metadata.db`

### 候选集优先检索

- 检索语义调整为：
  - 不填硬筛选：默认全库
  - 填了硬筛选且命中：先筛候选文献，再展开候选 chunks
  - 填了硬筛选但筛空：直接空结果，不回退全库
- 第二阶段进一步改为真正的候选子索引检索：
  - 先由元数据库得到候选 `paper_ids / chunk_ids / faiss_row_ids`
  - 再只在候选 `faiss_row_ids` 对应的向量子集上做搜索

### 调试与日志

- 日志可观察：
  - `candidate_source`
  - `candidate_papers`

## 2026-05-03

### 问答链路增强

- 在 `src/qa_agent.py` 增强了 intent 路由规则，减少 `paper_lookup` 和 `direct_answer` 的误分流
- 增加 query-aware chunk rerank，让检索更偏向和问题真正相关的段落
- 将 `paper_list` 路径的候选论文上限提高到至少 `10` 篇

### paper 聚合与回答生成调整

- 在 `src/aggregator.py` 改进 paper 聚合打分，不再只是简单累加 chunk 分数
- 增加 paper citation 覆盖逻辑，确保正文引用和右侧论文列表保持一一对应
- 将 fallback / 无引用时的补引用上限从 `5` 提高到 `10`

### 服务端引用过滤调整

- 在 `server.py` 保持“按正文引用过滤并重排 references”的逻辑，避免正文和列表脱钩
- 撤掉此前“直接强保留前 `10` 篇”的临时逻辑，统一回到按正文引用驱动列表展示

### prompt 改成归因式写法

- 将 `direct_answer` 和 `paper_list` 两条链路的 prompt 改成“基于材料归因写作”的版本
- 新 prompt 强调：重要判断要能对应材料，并使用 `[n]` 说明依据，不鼓励在段末集中堆编号
