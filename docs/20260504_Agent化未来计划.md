# AskMyZotero Agent 化未来计划

## 背景

当前 AskMyZotero 已经不只是“给一段 query 做向量检索再生成回答”的简单 RAG 结构。  
项目内部已经逐步形成了多个可组合的能力单元，例如：

- 元数据过滤
- query intent 识别
- query rewrite
- 检索调度
- paper 聚合
- 引用组织

当这些单元继续向上抽象，并由一个统一的控制层决定“下一步该做什么”时，系统形态就会从 RAG 进一步演化为 Agent。

一个典型目标场景是：

1. 用户提出一个信息不完整的问题。
2. 系统识别当前问题缺少关键槽位，不能直接回答。
3. 系统主动澄清，例如追问日期、对象、范围或口径。
4. 用户补充条件后，系统基于上下文补全结构化查询。
5. 系统再按计划调用过滤、检索、聚合、生成等能力。
6. 最终返回带依据、带引用、能解释过程的回答。

例如：

- 用户：`化学制品行业关注度排名第几？`
- 系统：这个问题缺少时间条件，不同日期的排名可能不同。你想看哪一天？
- 用户：`就看昨天的吧`
- 系统：
  - 继承上一轮中的“化学制品行业关注度排名”语义
  - 解析“昨天”为绝对日期
  - 先调用元数据过滤缩小范围
  - 再做检索与生成
  - 输出最终回答

这类能力目前还没有真正建立起来，下面整理当前缺失的关键功能。

## 当前缺失能力清单

### 1. 多轮对话状态与上下文记忆

当前问答主流程仍以单轮请求为主，缺少稳定的会话状态管理。  
如果用户第二句只说“就看昨天的吧”，系统很难自动继承上一轮问题中的主体、指标和约束。

需要补充：

- 会话级 `session_id` 与上下文存储
- 上一轮 query / filters / answer / candidate scope 的保留
- 用户追问、补充和改口时的上下文继承
- 多轮问答下的状态过期与清理机制

### 2. 指代消解与省略补全

Agent 必须理解：

- `昨天`
- `那个行业`
- `上面那篇`
- `第二篇论文`
- `还是刚才那个主题`

当前项目缺少这类指代解析与缺省参数补全能力。

需要补充：

- 对时间、对象、论文、集合条件的指代消解
- 对省略主语、宾语和筛选条件的自动补全
- 基于上一轮结构化结果的参数继承

### 3. 缺参识别与澄清追问

当前系统更擅长“收到 query 后尽量给出回答”，但 Agent 需要先判断“能不能答”。  
当 query 信息不足时，应优先追问而不是硬答。

需要补充：

- 判断当前问题是否缺少必要槽位
- 识别缺失的是时间、对象、范围、指标还是口径
- 输出机器可执行的澄清动作，而不是直接进入检索
- 澄清问题模板和追问策略

典型场景：

- 时间缺失：`哪天`
- 对象缺失：`哪篇`
- 范围缺失：`哪个 collection / 标签 / 作者范围`
- 比较项缺失：`和谁比`

### 4. 显式 Planner / Controller

现在的 `qa_agent.py` 更像“检索路由器”，还不是完整的 Agent 控制层。  
要实现多步决策，需要一个上层 planner 来决定当前该：

- 直接回答
- 先澄清
- 先做元数据过滤
- 先重写查询
- 先检索 chunk
- 先聚合到 paper

需要补充：

- 统一的 action schema
- 多步执行计划表示
- 中间结果保存
- 工具调用顺序控制
- 失败回退与兜底策略

### 5. 工具化封装

当前项目内部已经有很多“可被调度的能力”，但它们主要还是代码模块，而不是显式工具。  
如果要做 Agent，需要把这些能力整理成可调用工具层。

建议抽象的工具包括：

- `resolve_time_expression`
- `normalize_filters`
- `filter_metadata`
- `search_chunks`
- `aggregate_papers`
- `generate_answer`
- `request_clarification`

这样上层控制器才能自由编排，而不是只能走固定链路。

### 6. 时间语义理解

“昨天”“最近一周”“过去三年”“2021 年之后”这类表达，是 Agent 化过程中最常见也最实用的能力。  
当前项目已有显式 `date_range` 过滤，但自然语言时间理解能力还不完整。

需要补充：

- 相对时间解析
- 绝对日期归一化
- 与当前系统日期绑定
- 时间表达映射到 `date_range` 或更细粒度查询条件

注意：

- 文献库场景常见的是年份区间
- 如果未来扩展到日报、行业榜单、时序数据，可能需要支持“日级”而不只是“年级”

### 7. 结构化查询表示升级

现在的 `QueryBundle` 已经是不错的基础，但整体仍偏向检索链路组织。  
如果要承接 Agent 多轮规划，需要更完整的结构化表示。

建议未来支持的字段包括：

- `subject`
- `metric`
- `time_scope`
- `filters`
- `missing_slots`
- `clarification_needed`
- `planned_actions`
- `conversation_references`

这会成为连接“用户自然语言”和“工具调用动作”的中间层。

### 8. 证据充足性判断与拒答机制

Agent 不能只会“尽量回答”，还要会：

- 判断当前证据是否足够
- 判断当前问题是否存在多个合理解释
- 在条件不足时停止并追问
- 在检索为空时明确反馈原因

需要补充：

- 证据不足的判断规则
- 候选集为空的专门响应
- 澄清优先于胡乱回答的策略

### 9. 面向多轮的 API 与前端协议

当前 `/api/ask` 仍然更接近单轮问答接口。  
如果要支撑 Agent，前后端都需要引入会话和动作协议。

需要补充：

- 请求携带 `session_id`
- 返回包含 `need_clarification`
- 返回包含 `missing_slots`
- 返回包含 `next_action`
- 前端展示系统追问
- 前端继续在同一会话里追答

### 10. 可观察性与调试能力升级

一旦引入 Agent，多步规划会比当前 RAG 链路更难调。  
所以需要更完整的日志与可观测性支持。

建议补充：

- 每轮会话的结构化状态快照
- planner 决策日志
- 每步 tool call 入参 / 出参
- 澄清前后 query 变化记录
- 多轮上下文继承记录

## 推荐实现优先级

如果后续要以最低成本往 Agent 演进，建议先做下面四项：

### P1

- 多轮对话状态与会话记忆
- 缺参识别与澄清追问
- 时间表达解析
- 结构化查询表示升级

原因：

- 这是“单轮 RAG”到“会追问 Agent”的最低闭环
- 不需要一开始就把所有工具体系彻底重构

### P2

- 显式 planner / controller
- 工具化封装现有模块
- 面向多轮的 API 协议升级

原因：

- 这一层会决定系统是否真正具备“规划并调用工具”的能力

### P3

- 证据充足性判断
- 多步日志与调试可观测性
- 更复杂的指代消解和省略补全

原因：

- 这些能力会显著提升 Agent 稳定性，但可以放在第一轮闭环之后打磨

## 对当前代码结构的影响

后续如果推进 Agent 化，主要会影响这些模块：

- `src/qa_agent.py`
  从单轮问答主流程扩展为“会话感知 + 动作调度”的控制层。
- `src/domain_models.py`
  增加面向多轮和工具调用的结构化状态对象。
- `src/metadata_store.py`
  暴露更适合上层调度器调用的过滤接口。
- `src/aggregator.py`
  继续作为下游能力模块保留，但需适配 planner 输出。
- `server.py`
  增加会话接口、澄清返回结构和多轮 API 协议。

必要时可以新增：

- `src/conversation_state.py`
  管理多轮上下文状态。
- `src/planner.py`
  负责 action 选择与多步计划生成。
- `src/tools.py`
  统一封装可被 Agent 调用的工具能力。

## 阶段性目标定义

### 第一阶段目标

先做到：

- 用户第二轮补一句“昨天的”
- 系统能继承上一轮主题
- 系统能把“昨天”转成绝对时间
- 系统能继续完成过滤、检索和回答

只要做到这一步，项目就已经从单轮 RAG 明显迈向 Agent。

### 第二阶段目标

再做到：

- 能识别缺参并主动反问
- 能根据问题类型决定先过滤还是先检索
- 能把内部能力以工具方式调用

### 第三阶段目标

最终做到：

- 具备稳定的多轮任务执行能力
- 具备更强的可解释性和调试能力
- 具备从“被动回答”走向“主动协助”的 Agent 体验

## 备注

这份文档不是立即排期，而是对“项目未来从 RAG 走向 Agent”所缺失能力的集中整理。  
后续如果要正式落地，建议先从最小闭环切入，而不是一次性重构整条问答链路。
