# 全项目检索优化 Roadmap

日期：2026-04-23

适用范围：

- `src/parser.py`
- `src/scanner.py`
- `src/metadata_store.py`
- `src/qa_agent.py`
- `src/aggregator.py`
- `src/indexer.py`

## 1. 文档目标

本文档用于整理 AskMyZotero 当前检索链路中更普遍、跨模块的共性问题，并给出一份面向工程落地的检索优化 roadmap。

重点不是只解决单个问题类型，而是整体提升：

- 召回覆盖面
- 方法名/术语检索稳定性
- section-aware 检索能力
- 论文级排序质量
- 日志可解释性
- 后续功能扩展空间

## 2. 当前检索链路概览

当前项目的主检索链路大致为：

1. `scanner.py`
   - 扫描 PDF
   - 从 Zotero sqlite 提取元数据
2. `parser.py`
   - 清洗 PDF 文本
   - 按段落和 section 切 chunk
   - 生成 `section / chunk_type / context_window`
3. `indexer.py`
   - 为 chunk 建立向量索引
4. `metadata_store.py`
   - 建立 SQLite metadata 索引
   - 主要用于硬筛选
5. `qa_agent.py`
   - intent 分类
   - query rewrite
   - dense 检索
   - chunk 合并
6. `aggregator.py`
   - chunk 聚合成 paper
   - section 加权
   - 生成答案

## 3. 当前更普遍的问题

### 3.1 召回方式过于单一

当前主召回基本依赖 dense FAISS 相似度检索。

问题：

- 对术语别名不够稳
- 对缩写展开不够稳
- 对标题词和摘要词利用不足
- 对精确短语匹配支持弱
- 对方法名、模型名、缩写类 query 波动较大

后果：

- dense 命中对 embedding 表达很敏感
- 很容易出现“能搜到相关主题，但搜不到目标术语本体”的情况

### 3.2 检索对象主要还是 chunk，而不是 paper

当前主流程是：

- 先取 chunk
- 再把 chunk 聚合成 paper

问题：

- chunk-first 很容易被长文放大
- 同一篇论文出现大量 chunk 时会天然占优
- 综述、benchmark、主题高度相关文会经常霸榜

### 3.3 section-aware 主要是后处理，不是召回层能力

虽然 parser 已经做了：

- `section`
- `chunk_type`
- `context_window`

但在检索阶段，仍然主要是统一全文 dense search。

问题：

- 不能显式优先召回 `abstract / method / experiment`
- “方法类问题”无法天然偏向方法描述性 chunk
- section 信息更多只是排序附加项

### 3.4 metadata 能力没有充分进入检索层

当前 metadata store 主要用于：

- author 筛选
- tag 筛选
- collection 筛选
- year 筛选

问题：

- title 没有作为独立检索信号使用
- abstract 没有建立独立候选通道
- venue / DOI / canonical id 没有更深入服务召回
- metadata 更像过滤器，不像召回器

### 3.5 query 理解和规范化能力偏弱

当前 query 处理已有：

- intent 分类
- query rewrite
- 简单关键词抽取

但仍缺少：

- 方法别名扩展
- 缩写展开
- query pattern 结构化
- “使用 / 提出 / 比较 / 提到 / 综述”关系词识别

后果：

- 对 method query、comparison query、definition query 的稳定性都有限

### 3.6 缺少二阶段精排

当前常见流程是：

- 初召回
- 去重
- 聚合
- 回答

缺少一层真正的 rerank。

问题：

- 不能有效区分“相关”与“最适合回答”
- 不能区分“提到某方法”和“真正使用某方法”
- 不利于控制答案证据质量

### 3.7 论文多样性控制不足

目前去重主要在 chunk 级别完成。

问题：

- 缺少 paper-level quota
- 缺少 paper diversity rerank
- 缺少版本合并策略
- 候选列表容易被少数论文垄断

### 3.8 可观测性还不够细

虽然已有 prompt log 和 hard filter diagnostics，但对检索优化来说仍不够。

还缺：

- 每轮 query 新增多少论文
- 每个 section 的召回分布
- 标题命中 / 摘要命中 / 正文命中的占比
- top papers 的证据结构
- 相同工作不同版本是否被重复算入

## 4. 全项目优化目标

整体检索优化应围绕以下目标展开：

1. 提高术语与方法名检索稳定性
2. 提高候选论文覆盖面
3. 提高高价值 section 的证据命中率
4. 降低长文、综述、benchmark 的系统性偏置
5. 提升 paper 级排序可信度
6. 为不同 intent 提供更合适的检索路径
7. 增强日志诊断和可解释性

## 5. 核心优化方向

### 5.1 引入 Hybrid Retrieval

目标：

- 让 lexical 与 dense 互补

建议引入的检索信号：

- dense embedding retrieval
- title lexical retrieval
- abstract lexical retrieval
- optional full-text lexical retrieval

推荐收益：

- 方法名、缩写、专有词更稳
- query 中的显式术语不容易丢
- dense 负责语义泛化
- lexical 负责术语锚定

### 5.2 建立两阶段检索架构

建议统一改造成：

- Stage 1：宽召回
- Stage 2：精排

Stage 1 负责：

- 尽可能扩大相关候选集
- 不要求最终排序极准

Stage 2 负责：

- 判断哪些证据更适合回答
- 判断哪些论文更值得排前

### 5.3 将 section-aware 前移到召回层

建议把检索空间显式分为：

- `title / abstract`
- `method`
- `experiment`
- `body`
- `references`

不同问题类型使用不同 section 配比：

- `definition`
  - 偏 `title / abstract / introduction`
- `fact_qa`
  - 偏 `abstract / method / experiment`
- `comparison`
  - 偏 `method / experiment / conclusion`
- `paper_lookup`
  - 宽 paper recall + 窄方法证据
- `survey`
  - 偏 `abstract / introduction / conclusion`

### 5.4 强化 metadata-driven retrieval

建议让 metadata 不只负责 hard filter，还参与 soft retrieval：

- 标题命中作为单独得分通道
- abstract 建立单独检索入口
- author / tag / venue 作为候选增强信号
- canonical paper id 用于版本合并和 paper 去重

### 5.5 建立 query normalization 层

建议新增统一 query normalization / expansion 模块，负责：

- 缩写展开
- 方法别名表扩展
- 连字符与大小写规范化
- 常见问句模式归一
- 意图相关关系词抽取

例如：

- `RAG`
- `retrieval augmented generation`
- `retrieval-augmented generation`

应在内部统一为同一组目标术语。

### 5.6 重构 paper-level aggregation

建议将当前“chunk 分数直接线性累加”的聚合方式，改成：

- top-N capped sum
- bucketed fusion
- diminishing return
- evidence quality aware scoring

避免：

- 一篇论文仅因 chunk 多而霸榜

### 5.7 增加 rerank / evidence selection 层

推荐从轻到重逐步引入：

1. 规则 rerank
2. section-aware rerank
3. metadata-aware rerank
4. cross-encoder rerank
5. LLM-based relation rerank

### 5.8 提升日志与评估能力

建议对每个 query 输出结构化诊断：

- 初召回 chunk 数
- 初召回 paper 数
- 各 query / bucket 的新增 paper 数
- section 分布
- top paper chunk share
- title / abstract / body 命中占比
- 最终回答中实际引用的 paper 数

## 6. 分阶段实施 Roadmap

## Phase 0：基线与诊断完善

目标：

- 先让问题可观测、可复现、可比较

Checklist：

- [ ] 建立固定检索测试集，覆盖：
  - 方法查询
  - 定义查询
  - 比较查询
  - survey 查询
  - paper_lookup 查询
- [ ] 给每类 query 记录基线输出
- [ ] 新增统一检索指标：
  - `retrieved_chunk_count`
  - `unique_paper_count`
  - `top_5_paper_chunk_share`
  - `top_10_paper_chunk_share`
  - `used_evidence_count`
  - `final_answer_paper_count`
- [ ] 在日志中补充：
  - 每轮 query 的新增 paper 数
  - 各 section 召回分布
  - 同一工作重复版本统计

## Phase 1：低成本高收益优化

目标：

- 在不大改架构的前提下，先缓解当前最显著问题

Checklist：

- [ ] 为 `paper_lookup` 和其他多文献类 query 增加 paper-level quota
- [ ] 将 paper 聚合由线性累加改为 `top-N capped + decay`
- [ ] 降低标题词命中带来的过强偏置
- [ ] 引入简单的版本合并策略：
  - DOI 优先
  - title normalization 辅助
- [ ] 在现有 dense 检索基础上，增加 title / abstract 的轻量 lexical 加权
- [ ] 针对常见方法名建立最小版 alias 词表

## Phase 2：引入 Hybrid Retrieval

目标：

- 从单路 dense 升级为多路召回

Checklist：

- [ ] 增加 title lexical index
- [ ] 增加 abstract lexical index
- [ ] 设计 lexical 与 dense 的分数融合策略
- [ ] 对不同 intent 设置不同融合权重
- [ ] 为方法名 / 模型名 /缩写类 query 单独调权
- [ ] 验证 hybrid 后对方法名检索稳定性的提升

## Phase 3：重构为两阶段检索

目标：

- 让“宽召回”和“精排序”解耦

Checklist：

- [ ] Stage 1：构建宽候选召回层
- [ ] Stage 1 输出 paper candidate set 与代表 chunk
- [ ] Stage 2：增加精排层
- [ ] 精排时引入：
  - section-aware score
  - title / abstract evidence score
  - metadata boost
- [ ] 对不同 intent 输出不同 top-N 策略

## Phase 4：section-aware 检索前移

目标：

- 不只在聚合时用 section，而是在召回时就利用 section

Checklist：

- [ ] 为 chunk 建立 section bucket
- [ ] 支持按 section 召回：
  - abstract-first
  - method-first
  - experiment-first
- [ ] 为每个 intent 定义 section retrieval profile
- [ ] 让 evidence selection 优先从高价值 section 中抽取
- [ ] 降低 references 对主答案检索的干扰

## Phase 5：增强 query normalization 与 query planning

目标：

- 提升 query 到检索动作的映射质量

Checklist：

- [ ] 建立 query normalization 模块
- [ ] 支持缩写展开
- [ ] 支持方法别名扩展
- [ ] 支持 query pattern 结构化
- [ ] 区分：
  - 使用
  - 提出
  - 比较
  - 提到
  - 综述
- [ ] 将自然语言 rewrite 升级为“内部检索视角 planning”

## Phase 6：增强 paper-level rerank 与多样性控制

目标：

- 让结果更像“高质量文献集合”，而不是“重复 chunk 排序”

Checklist：

- [ ] 增加 paper diversity rerank
- [ ] 对同类论文过度集中做适度衰减
- [ ] 对综述、benchmark、tutorial 做显式标记
- [ ] 增加不同版本合并后的 paper-level 去重
- [ ] 控制单篇论文最终参与回答的 chunk 数

## Phase 7：引入更强精排器

目标：

- 在基础结构稳定后，再引入更强模型

Checklist：

- [ ] 评估 cross-encoder reranker 的接入成本
- [ ] 评估 LLM-based relevance / relation rerank 的收益
- [ ] 评估是否按 intent 走不同 rerank pipeline
- [ ] 为精排器增加缓存与批处理策略

## 8. 推荐优先级

如果按投入产出比排序，建议优先顺序如下：

1. `Phase 0`
   - 完善基线与日志
2. `Phase 1`
   - 先修 paper 聚合与 paper 多样性问题
3. `Phase 2`
   - 引入 hybrid retrieval
4. `Phase 4`
   - 将 section-aware 前移到召回层
5. `Phase 5`
   - 增强 query normalization 与 query planning
6. `Phase 3`
   - 完整重构为两阶段检索
7. `Phase 6`
   - paper-level rerank 与多样性控制加强
8. `Phase 7`
   - 引入更强精排器

## 9. 阶段性验收标准

优化过程建议至少观察以下结果是否改善：

- [ ] 方法名 / 缩写类 query 结果更稳定
- [ ] `paper_lookup` 的 `unique_paper_count` 上升
- [ ] `top_5_paper_chunk_share` 下降
- [ ] 高价值 section 的 evidence 比例上升
- [ ] survey / benchmark / 长文的系统性霸榜现象下降
- [ ] 最终回答引用的论文覆盖数增加
- [ ] debug 日志能说明“为什么召回到这些结果”

## 10. 一句话总结

当前项目的主要瓶颈不是向量模型本身，而是检索结构还比较单一：

- 单路 dense chunk 检索
- metadata 只做硬筛选
- section-aware 主要停留在排序层
- paper 聚合容易被重复 chunk 放大

全项目更合理的优化方向是：

- `hybrid 多路召回`
- `two-stage 检索与精排`
- `section-aware 前移`
- `paper-level 融合与多样性控制`
- `query normalization + retrieval planning`
