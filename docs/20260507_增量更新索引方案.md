# 增量更新索引方案

## 总目标

当 Zotero storage 中 PDF 新增、删除或修改时，尽量避免整库重建，只处理发生变化的文件。

整体分三阶段推进：

- Phase 1：显式增量更新，用户点击同步或命令行触发后，只处理新增、修改、删除的 PDF。
- Phase 2：索引压缩与健康检查，解决 FAISS 中 tombstone 旧向量长期累积的问题。
- Phase 3：自动化同步策略，在合适时机自动检测 Zotero 变化并提醒或执行增量更新。

## Phase 1：显式增量更新

### 目标

Phase 1 采用低风险策略：

- 新增 PDF：解析、切块、向量化并追加到 FAISS。
- 修改 PDF：旧 chunk 从 active metadata 中移除，重新解析、切块、向量化并追加。
- 删除 PDF：旧 chunk 从 active metadata 中移除。
- FAISS 暂不物理删除旧向量，通过 metadata.db active 集合和检索过滤实现 tombstone。

### 为什么先不物理删除 FAISS 向量

LangChain FAISS 的 `index_to_docstore_id`、docstore 和 FAISS row id 需要保持一致。直接删除中间向量容易导致 row id 映射错位。

Phase 1 先使用 tombstone：

- metadata.db 只写入当前 active splits。
- 检索后过滤不在 metadata.db active chunk 集合中的旧 chunk。
- 新 chunk 直接 append 到 FAISS。
- 后续当 inactive 向量比例变高时，再做 compact/rebuild。

### 当前流程

```text
scan PDFs
-> load old manifest
-> build current manifest records
-> diff added / modified / removed / unchanged
-> load splits cache
-> drop removed/modified old splits from active cache
-> parse added/modified PDFs
-> append new splits to FAISS
-> save updated splits cache
-> rebuild metadata.db from active splits
-> save new manifest
```

### 当前触发入口

- 命令行：`python main.py --incremental`
- 前端：主页面右上角“同步”按钮，调用 `/api/sync`
- 后端 API：`POST /api/sync` 启动后台任务，`GET /api/sync/status` 查询状态

### 需要注意

- Phase 1 暂不做 metadata-only 更新优化；如果文件或记录变化导致 manifest diff 认为需要更新，会按 modified 路径重切和重嵌入。
- 旧向量仍存在于 FAISS 文件中，但不会通过 metadata.db active 过滤进入问答。
- 长期运行后建议增加 compact rebuild：当 FAISS 总向量数明显大于 active chunk 数时，提示用户做一次全量压缩重建。

### 验证方式

- 新增一个 PDF 后重启/初始化，确认只解析新增文件。
- 删除一个 PDF 后重启/初始化，确认该 paper 不再出现在检索结果。
- 修改一个 PDF 后重启/初始化，确认旧 chunk 不再 active，新 chunk 可检索。
- 检查 prompt log 中 `hard_filter_diagnostics` 的 inactive 过滤计数。

## Phase 2：索引压缩与健康检查

### 目标

Phase 1 使用 tombstone 后，FAISS 文件中会保留旧向量。短期这是安全的，但长期会让索引文件膨胀，并增加检索后过滤成本。Phase 2 的目标是让系统能判断索引是否“虚胖”，并提供安全的 compact rebuild。

### 方案

- 增加索引健康统计：
  - FAISS docstore/vector 总量。
  - metadata.db active chunk 数量。
  - inactive chunk 估算数量。
  - inactive ratio，例如 `(faiss_total - active_total) / faiss_total`。
- 增加 compact rebuild 能力：
  - 只使用当前 active splits cache 重新构建一个新的 FAISS。
  - 新索引先写入临时目录，例如 `faiss_index.compacting/`。
  - 构建完成并校验后，再原子替换现有 `faiss_index/`。
  - 替换失败时保留旧索引，不影响问答。
- 增加前端提示：
  - 当 inactive ratio 超过阈值，例如 30% 或 40%，在同步完成后提示“建议压缩索引”。
  - 压缩索引应当和普通同步分开，避免用户误以为每次同步都会做重活。

### 触发条件

- 手动触发：设置页或主页面提供“压缩索引”入口。
- 自动建议：同步完成后如果 inactive ratio 超过阈值，只提示，不自动执行。
- 命令行触发：增加类似 `python main.py --compact-index` 的显式命令。

### 验证方式

- 构造新增、修改、删除后产生 inactive 向量的场景，确认健康统计能识别差异。
- 执行 compact 后，确认 FAISS 总量接近 active chunk 数。
- compact 后执行同一组问答，确认引用和检索结果仍可用。
- 模拟 compact 中途失败，确认旧索引没有被破坏。

## Phase 3：自动化同步策略

### 目标

Phase 1 需要用户手动点击同步。Phase 3 的目标是减少用户记忆成本，让系统能够在合适时机发现 Zotero storage 的变化，并自动提醒或自动同步。

### 方案

- 启动时轻量检测：
  - 服务启动后只做 manifest diff，不直接嵌入。
  - 如果发现新增、修改、删除，前端提示“检测到文献库变化，是否同步？”。
- 前端定时检测：
  - 增加轻量接口，例如 `GET /api/sync/check`。
  - 只返回是否有变化、变化数量，不执行索引更新。
  - 前端可以每隔一段时间检测一次，例如 5 到 10 分钟。
- 可选自动同步：
  - 只在用户开启设置项后启用。
  - 适合小规模变化，例如新增 PDF 数量低于阈值。
  - 大规模变化仍建议只提示，由用户确认。
- Zotero metadata 变化处理：
  - Phase 1 主要关注 PDF 文件变化。
  - Phase 3 可进一步比较 Zotero sqlite 中的 title/authors/tags/collections/year 等 metadata hash。
  - metadata-only 变化不需要重新嵌入，只需要更新 splits cache metadata 和 metadata.db。

### 触发策略建议

- 默认策略：只提醒，不自动同步。
- 安全自动策略：只对少量新增 PDF 自动同步。
- 保守策略：检测到修改或删除时只提醒，因为这会涉及 tombstone 和引用可见性变化。

### 验证方式

- 启动服务前新增 PDF，确认启动后前端能提示变化。
- 修改 Zotero metadata 但不改 PDF，确认 metadata-only 路径不会重嵌入。
- 定时检测不会阻塞问答，也不会频繁扫描造成明显卡顿。
- 自动同步失败时，前端能看到失败原因，原有索引仍可继续问答。
